\section{Experimental Methodology}
\label{sec:experimental_methodology}
This chapter evaluates five order book implementations against three core
research hypotheses (\ref{hyp:h1}, \ref{hyp:h2}, and~\ref{hyp:h3}). The benchmarking environment is isolated via
thread-pinning to the x86\_64/WSL2 host (Ryzen 7600X) to ensure
reproducible nanosecond-level precision. Given that WSL2 is a virtualised, 
Hyper-V-managed utility VM~\citep{microsoft_wsl2_arch}, there is an inherent 
"virtualisation tax" and network overhead. 

Crucially, the act of measurement itself introduces an "Observer Effect" 
latency. On the host environment, the instrumentation logic for capturing 
operation-level breakdowns adds an overhead of approximately 100--150~ns to 
the \texttt{Total\_Mean} latency. While this tax is applied equally across all 
implementations, we formally verified its consistency by measuring the 
overhead of back-to-back instrumentation calls (the ``timing bracket'' 
consisting of a start and end timestamp) across $10,000$ iterations. 
This revealed a standard deviation of less than 10~ns, confirming 
that while instrumentation adds a constant base-latency, it does not 
introduce significant measurement noise or variance that would skew 
the comparative validity of different order book variants. 
Consequently, while absolute algorithmic latencies for the \texttt{ArrayOrderBook} 
(reported as ~214~ns) are likely closer to ~60--100~ns in production, the 
cross-implementation performance trends remain robust.

To isolate systemic and algorithmic latency, the benchmarking suite incorporates 
two distinct modes. \textbf{Direct mode} focuses on isolating algorithmic 
performance by using thread-pinning and removing transport overhead, thereby 
adhering to the isolation standard defined in~\NFR{3}. This configuration 
allows for the capture of performance trends within the order book 
implementations themselves with minimal external interference. Conversely, \textbf{Gateway mode} provides a realistic evaluation by incorporating network 
transport, queueing, FIX parsing, and matching engine processing. This offers a 
shadow-to-engine view of the end-to-end pipeline in accordance with~\NFR{4}.

When collecting data, we capture significant statistics for four core operations: 
\textbf{insert, cancel, lookup, and matching}. For each of these, we calculate 
a set of metrics, including \textbf{mean latency, standard deviation, 
P99 tail latency, and throughput}, adhering to~\NFR{6}. In gateway and MPSC 
modes, we also decompose these latencies into \textbf{network, queue, and engine 
processing} components to isolate performance bottlenecks. An important 
attribute of our data is to make it reproducible; this is achieved by a 
well-documented benchmarking suite that generates a canonical dataset every time 
it is run, adhering to~\NFR{5}. To guarantee scientific validity, we rerun our 
benchmarks multiple times per configuration, where a configuration is defined
by \texttt{Mode/Scenario/Book} (and \texttt{Producers} for MPSC rows). We then
use the run set for that configuration as the basis for statistical
summary. In this dataset, direct-mode configurations use $n=30$
samples, gateway configurations use $n=20$ samples, and MPSC
configurations use $n=30$ samples for each
\texttt{Scenario/Book/Producers} setting (with producers $\in\{1,2,4,8\}$);
each sample run still processes the full per-run order volume.
We use a larger $n$ for direct mode because those runs are cheaper to execute,
while gateway runs include full network/FIX/queue stages and therefore take
longer per replicate; this balances precision against total time taken.
These replicate counts provide stable summary statistics for this chapter's
hypothesis-level analysis and narrow uncertainty intervals around the key
cross-implementation comparisons.

To quantify estimation uncertainty, we report \textbf{95\% bootstrap percentile
intervals}~\citep{efron1994introduction}. For each implementation cell, we draw
$B = 10{,}000$ resamples with replacement from the $n$ observed replicate means,
and take the 2.5th and 97.5th percentiles of the resulting distribution of
bootstrap means $\{\bar{x}_1^*, \ldots, \bar{x}_B^*\}$:
\[
  \bigl[B_{\,0.025}^{*},\; B_{\,0.975}^{*}\bigr].
\]
This approach makes no distributional assumptions — an important property for
latency data, which routinely exhibits the heavy right-hand tails caused by
cache misses and OS scheduling jitter. For reading the results when two
implementations show non-overlapping 95\% bootstrap confidence intervals
within the same scenario, that is a strong practical signal that the 
observed latency gap is not simply run-to-run noise. Where we need a 
formal verdict on whether a gap
is statistically significant, we use the paired sign-flip permutation test
described fully in Section~\ref{sec:statistical_methodology}.

While runs were conducted sequentially, the use of identical warmup 
procedures and the massive separation of $p$-values ensure our results 
reflect real architectural differences rather than random system jitter. 
This is explicitly supported by the verified timing-bracket stability 
documented in Section~\ref{sec:experimental_methodology}, showing that 
residual noise is at least an order of magnitude smaller than the observed signals.

Finally, to evaluate these findings, we refer to the original research hypotheses
(\ref{hyp:h1}, \ref{hyp:h2}, and~\ref{hyp:h3}) as our primary analytical framework. 
To maintain structural rigour, each performance claim in the subsequent sections is 
mapped to its corresponding hypothesis and any relevant Non-Functional (NFR) or Functional (FR) requirement IDs.

\section{Functional Validation}

\subsection{Core Logic and Semantic Parity}
To verify that our order books follow semantic parity~(\NFR{1}), we used 
parametrised contract testing~\citep{fowler_contract_testing} (\FR{6}). Rather than 
testing our order books individually, we applied the same tests to each 
implementation in the form of a contract. We achieved this by using Google Test's 
parametrised suites~\citep{googletest_param}, ensuring each order book was 
verified against identical market scenarios (including happy paths, partial 
fills, and edge cases). This is critical as it guarantees that any subsequent 
performance differences observed are due to architectural efficiency rather than 
logical differences, ensuring fairness in our comparisons~(\NFR{2}).

This approach follows the \textbf{Ports and Adapters architecture}~\citep{cockburn_hexagonal}, 
using contracts that verify compliance with the \textbf{Liskov Substitution Principle}~\citep{liskov1987}. 
Beyond validating correctness, this strengthens code maintainability. Utilising the 
LLVM coverage toolchain~\citep{llvm_cov}, the project achieved 100\% coverage 
across branches, lines, regions, and functions in the core order book logic 
(summarised in Table~\ref{tab:code_coverage_results}), proving that the 
benchmarking results are derived from fully validated and semantically equivalent components.

To evaluate the system's performance limits across different conditions, we integrated 
a range of market scenarios. These act as a rigorous functional test, revealing 
how each order book performs under varying loads. These scenarios include: 
\begin{itemize}
    \item \textbf{Full Aggressive Matches}: To verify the matching logic's ability 
    to correctly match and remove all pending orders across multiple price levels.
    \item \textbf{Deep-Book Scenarios}: To test the data structure's efficiency in 
    maintaining price-time priority as both the number of price levels (market 
    depth) and the number of orders per level (queue depth) increase.
    \item \textbf{Invalid Entry Handling}: To verify that zero-quantity or illegal 
    price orders are safely rejected without corrupting the internal state.
\end{itemize}
We compared all five implementations against a \textbf{golden reference} derived 
from the \texttt{MapOrderBook} implementation. As the most straightforward translation of 
limit order book logic, it serves as the benchmark to ensure that 
the more complex, high-speed implementations maintain identical logical behaviour. 

To ensure the statistical validity of the most complex test case, the 
\textbf{mixed} scenario, we define three price-proximity tiers relative to the 
market midpoint. These consist of \textbf{near-spread} activity (orders within 
$\pm$6 ticks of the mid-price), \textbf{deep share} ($\pm$10 to $\pm$50 ticks), 
and \textbf{extreme share} (outliers at $\pm$400 to $\pm$600 ticks), where a 
`tick' represents the minimum price increment of the exchange. As shown in 
Figures~\ref{fig:mixed_profile_dist} and~\ref{fig:mixed_profile}, this configuration is validated by capturing the 
branch probabilities within the order generator to confirm the workload 
maintains a heterogeneous distribution rather than collapsing into a single 
regime~(\NFR{1}). This level of transparency in the benchmark generation ensures 
that all subsequent performance claims are strictly bounded by known evidence, 
adhering to~\NFR{9}.

\subsection{Scaling and Stress Testing}
While the primary architecture uses a custom-built, wait-free \textbf{Single-Producer 
Single-Consumer (SPSC)} queue for learning and simplicity, we also needed to 
test how the system behaves under producer contention. To achieve this, we 
integrated an open-source \textbf{Multi-Producer Multi-Consumer (MPMC)} queue 
from the \textbf{Moodycamel} library~\citep{moodycamel_concurrentqueue}. By 
running this queue in a single-consumer configuration, we were able to evaluate 
how the system behaves under producer contention from multiple concurrent 
clients pushing into the matching engine. Both queue implementations were 
subjected to the same functional validation to ensure that the choice of 
transport did not affect the final order book state.

The \texttt{FIXParser} was also verified to ensure it behaves correctly during 
fragmentation (where the network stream is split into multiple packets) by 
successfully reassembling messages. The queue was tested for back-pressure 
handling; if the 1024-deep queue reaches capacity, a \texttt{push()} call will 
fail, with the expected behaviour being that the client is alerted rather than 
the system crashing. Finally, Moodycamel's MPMC queue was integrated to ensure 
that the system maintains stability and does not significantly degrade in 
performance when multiple concurrent clients are querying the engine at the 
same time.

\subsection{Gateway Simulation Boundaries and Validation}
To ensure the end-to-end performance results are technically sound, we 
will define the boundaries of the Gateway simulation. 
As identified in Chapter~\ref{chap:implementation}, the benchmark utilises a \textbf{Loopback (AF\_INET)} 
configuration where the \texttt{MockClient} and \texttt{TCPOrderGateway} 
communicate via the local interface (\texttt{127.0.0.1}). 

This setup was verified through system-level profiling—using \texttt{strace} 
to monitor syscall boundaries and \texttt{perf} to observe kernel-space stack 
traversal—confirming that the workload correctly exercises the 
\textbf{Linux TCP/IP machinery}~\citep{benvenuti_networking}. While 
the physical wire is bypassed, the "software networking tax" (system calls, 
\texttt{sk\_buff} allocation, and context switching jitter) remains fully 
present in the "Net" component measurements.
However, because the client and server reside on the same physical host, 
this environment eliminates the latency of physical network media (cables) 
and NIC hardware processing (e.g., PCIe bus latency). 
Consequently, the absolute results recorded in the "Net" component provide a 
\textbf{software-isolated baseline} for system latency, effectively 
differentiating the "software networking tax" from the latency of 
physical transport media.



\section{Throughput Results}
\label{sec:throughput_results}
This section evaluates hypotheses \ref{hyp:h1} and \ref{hyp:h3} by analysing system 
throughput in direct and gateway modes.

The throughput results in direct mode work to verify~\ref{hyp:h1}, which
claims that order books with index-based layouts will be faster than those with
search trees or sorted vectors. We can see evidence for this claim by observing
the greatest direct throughput of 4,767,353~ops/s achieved by the
\textbf{ArrayOrderBook}. It remains clearly ahead of the next tier
(\textbf{PoolOrderBook} at 1,663,118~ops/s and \textbf{MapOrderBook} at
1,552,168~ops/s), while \textbf{VectorOrderBook} is lowest at 66,666~ops/s.
These direct-mode values are averaged across seven scenarios
(tight spread, fixed levels, dense full, sparse extreme, worst case FIFO, mixed,
and high cancellation), each of which exercises a mix of insert, cancel,
lookup, and match operations, adhering to~\NFR{6}.
Although \textbf{ArrayOrderBook} wins here, it is not without its downsides.
When using index-based order books, the implementer is forced to define a fixed
range of valid prices and a discrete tick size. Additionally, logic must be
provided to handle orders whose prices fall outside the maximum and minimum
values of the order book, or that do not fall exactly on a price level.
These restrictions may limit the flexibility and accessibility of the exchange 
compared to standard pointer-based designs.
The second-place \textbf{PoolOrderBook}, by contrast, imposes no such
price-range constraints. Its competitive throughput can be attributed to
its custom memory pool, which eliminates dynamic heap allocation on
the critical path and reduces allocator overhead compared to the
pointer-chasing traversal of \textbf{MapOrderBook}.
For these best-versus-worst direct comparisons, the 95\% bootstrap confidence intervals are clearly
separated in the generated summaries, so the performance gap is unlikely to be
explained by run-to-run noise.

\paragraph{Statistical Methodology.}
\label{sec:statistical_methodology}
To test whether observed latency gaps reflect genuine structural differences 
rather than run-to-run noise, we apply a \textbf{paired sign-flip permutation 
test}~\citep{pitman1937significance} - a non-parametric approach that makes 
no distributional assumptions, which is important given the heavy right-hand 
tails common in low-latency data. Multiple comparisons across the four 
implementations are controlled via the \textbf{Holm-Bonferroni step-down 
procedure}~\citep{holm1979simple}. For each (Mode, Scenario) combination, the 
comparison family $m=4$ consists of the four alternative implementations 
compared against the \texttt{ArrayOrderBook} baseline. All permutation tests 
are \textbf{paired by replicate ID}, ensuring each implementation processes 
the exact same sequence of orders to guarantee a fair ``apples-to-apples'' 
comparison.

Across all primary scenarios, the results provide overwhelming evidence against 
the null hypothesis. In the mixed regime, comparing the ArrayOrderBook to the 
MapOrderBook yields a decisive result ($p < 0.0001$). A similarly extreme 
separation is observed in the \texttt{dense\_full} scenario against the map 
alternative ($p < 0.0001$). Even in the least computationally intensive 
scenario (\texttt{tight\_spread}), the array implementation maintains a clear 
statistical lead ($p < 0.0001$). This formal analysis confirms that the 
memory-layout optimisation in the array implementation delivers a genuine, 
statistically significant latency reduction that cannot be attributed to 
environment jitter.

The gateway mode results support~\ref{hyp:h3}. In a realistic exchange
with overheads in the form of the network and queues, these additional latencies
mask the performance differences between order books. In direct mode (see Table~\ref{tab:throughput_results}), 
the throughput spread across all implementations is extreme, ranging from 
4,767,353~ops/s for the \texttt{ArrayOrderBook} to 66,666~ops/s for the 
\texttt{VectorOrderBook}.
In gateway mode (for the three representative scenarios), the spread is 
significantly compressed (476 vs 289~ops/s), with non-vector implementations 
still clustered more tightly than in isolated mode. 

We restricted gateway benchmarks to three representative scenarios 
(tight\_spread, mixed, and dense\_full) to focus our available compute-time on 
realistic simulations for significant market conditions. While 
\texttt{sparse\_extreme} showed the largest direct-mode performance gap, 
we excluded it from gateway analysis as the "masking effect" is more 
systematically demonstrable in high-density workloads where queue 
pressure is maximal. Given that 
each gateway run requires full-stack network and queue processing, this focused 
selection balances campaign duration with the statistical precision required for \ref{hyp:h3}.

\section{Latency Analysis - Direct Mode}

This section provides a deeper analysis of the per-operation latencies across all
order book implementations, strictly in direct mode to isolate data structure
behaviour from transport overhead. The results are examined across a range of
market scenarios to highlight the strengths and weaknesses of each design
under different types of market activity. These observations are then used to
evaluate~\ref{hyp:h1} and~\ref{hyp:h2}.


Table~\ref{tab:direct_mean_latency} reports per-operation latency for the
\texttt{mixed} scenario and shows the same ordering trend: the
\textbf{ArrayOrderBook} remains the fastest implementation, with lower insert,
cancel, lookup, and match costs than the other books in this workload.
The reason the array data structure yields better performance over the other
order book structures is consistent with mechanical sympathy~\citep{thompson2011disruptor}. Because elements
in an array are contiguous, they are more cache-friendly~\citep{patterson2013cod}. During a cache hit,
the processor is able to return contiguous elements in the same cache lines,
whereas tree structures such as \textbf{MapOrderBook} do not exhibit this
behaviour since each node in the tree may reside at an arbitrary heap address.
To provide empirical support for these findings, absolute performance counters were
recorded in direct mode using Linux \texttt{perf stat} for pinned, single-core
benchmarks~\citep{linux_perf}. L1 data-cache and branch prediction metrics were
collected across the \texttt{mixed} and \texttt{dense\_full} scenarios.
In the \texttt{mixed} scenario, \textbf{ArrayOrderBook} recorded 843,745 L1 data-cache
misses and a 1.52\% branch miss rate, compared to 1,392,193 misses and a 1.58\%
branch miss rate for \textbf{MapOrderBook}.
The performance gap is most evident in the \texttt{dense\_full} scenario:
\textbf{VectorOrderBook} recorded a severe 144,837,499 L1 misses and
executed 366,302,717 branch instructions (12x the volume of \textbf{ArrayOrderBook}),
consistent with the overhead of O(n) sorted insertion.
Furthermore, while the \textbf{PoolOrderBook} benefits from cache locality, it
exhibits the highest branch miss rate of 7.36\% in the \texttt{mixed} scenario,
explained by the additional conditional logic required for custom pool management.
These measured results indicate that the \textbf{ArrayOrderBook}'s advantage
stems from a combination of superior cache hit-rates, execution-path simplicity,
and a branch-predictor-friendly
execution path, in accordance with~\NFR{8}.

By analysing performance across different market conditions, we identify where each 
architecture excels. Figure~\ref{fig:latency_mean_heatmap} shows that 
\textbf{ArrayOrderBook} is the fastest implementation in all seven direct-mode
scenarios, not just in deep-book stress cases. This is stronger than our earlier
expectation that map-like structures might match array performance in shallow
regimes such as \texttt{tight\_spread} or \texttt{fixed\_levels}. Figure~\ref{fig:latency_slowdown_vs_best_heatmap}
confirms this: in those scenarios, non-vector books are still around 2.8x--3.3x
slower than array. To stress structural limits further, we engineered
\texttt{dense\_full} with 2,000 active price levels; this is where the largest
separation appears.

    \caption{Direct-mode mean latency heatmap across scenarios and order book 
    implementations. Array is the fastest implementation in all seven

The \textbf{VectorOrderBook} exhibits the largest degradation in latency, increasing
from 658.52~ns to 15,339.73~ns in the \texttt{dense\_full} scenario. This is likely due
to the sorted vector having to shift a large number of elements in memory during
frequent insertions. Additionally, this shifting causes a cascade of L1 data-cache
misses - a mechanism reinforced by the Linux \texttt{perf} results,
which recorded 144,837,499 L1 misses and executed 366,302,717 branch instructions for
the vector implementation. To put this in perspective, under the exact same \texttt{dense\_full}
workload, the vector implementation suffered \textbf{216x more L1 cache misses} and
executed \textbf{12x more branch instructions} than the \textbf{ArrayOrderBook}
(which recorded only 670,762 misses and 30,567,574 branches). These results 
strongly indicate that sorted-vector designs
in HFT are unscalable as the order book grows, especially in deep books. A strong trend
among all scenarios is the resilience of the \textbf{ArrayOrderBook}. It stays between
206~ns and 224~ns in all cases, likely due to the $O(1)$ direct index mapping. As a
consequence of this, the algorithmic complexity of operations such as inserting,
cancelling, and looking up orders is identical whether there are 10 or 10,000 price
levels. This provides strong empirical support for~\ref{hyp:h1}.
The vector implementation remains behind map and pool in most scenarios, and
then collapses in \texttt{dense\_full}.

While the \textbf{VectorOrderBook} is included in the mean latency summaries, it 
is excluded from the relative slowdown heatmap in 
Figure~\ref{fig:latency_slowdown_vs_best_heatmap}. This is a deliberate 
aesthetic and analytical choice: since the vector implementation exhibits a 75x 
degradation compared to the array baseline in \texttt{dense\_full}, its inclusion 
for all other scenarios, masking the subtle performance trade-offs between the 
more competitive implementations. By excluding the vector outlier, the 
high-resolution differences between index-based and tree-based structures become

\subsection{Architectural Mechanisms: Pool and Hybrid}
\label{subsec:pool_hybrid_mech}
Two novel implementations, the \textbf{HybridOrderBook} and the \textbf{PoolOrderBook},
were engineered in an attempt to mitigate the architectural limitations of the standard
data structures. Rather than relying on a single collection type, these order books employ
more complex internal logic to optimise for specific hardware characteristics.

The \textbf{HybridOrderBook} was designed as an improvement over the standard map by
employing a tiered architecture: a highly optimised "hot storage" and a slower
"cold storage." It aims to achieve the unbounded flexibility of a map - removing the
strict price-range limitations of the array - while mimicking array-like performance
for the most actively traded prices. The hot storage is implemented as a
capacity-bounded \texttt{std::vector} capped at the top 20 price levels. Because the
vector never exceeds 20 elements, memory-shifting during insertions is practically
instantaneous and fits entirely within the L1 cache. The data supports this
design trade-off. As seen in Table~\ref{tab:direct_mean_latency}, the Hybrid book
is still slower than the \textbf{ArrayOrderBook} in this mixed
operation breakdown (for example, 71.3~ns vs 31.7~ns on lookup), and it is also
slower than \textbf{MapOrderBook} in all four mixed operations shown in the same table. In other
words, the expected hot-storage advantage is not observed as a net win over \textbf{MapOrderBook}
in this dataset. A key reason is that in shallow books, \textbf{MapOrderBook} traversal depth is 
already low. Consequently, the overhead of hybrid's internal routing (checking 
tiers and handling boundary logic) outweighs the memory benefit of the 20-level hot tier.
 This also helps explain the higher
run-to-run spread observed for hybrid in \texttt{mixed} and
\texttt{high\_cancellation}: cancellation-heavy flows and heterogeneous order placement
increase tier-crossing frequency, so branch outcomes in the routing path become
less predictable on average. However, its
performance is sensitive to market depth.
In shallow scenarios, \textbf{HybridOrderBook} remains in the same non-array
performance band as \textbf{MapOrderBook} and \textbf{PoolOrderBook}
(for example, \texttt{tight\_spread}: 678 vs 639 vs 596~ns;
\texttt{fixed\_levels}: 689 vs 664 vs 609~ns),
but in the \texttt{dense\_full} scenario
its mean latency degrades to 2,143.29~ns. As market depth expands, a majority of orders
are pushed out of the 20-level vector and into the slower cold \texttt{std::map},
subjecting them to tree-traversal overhead.

In contrast, the \textbf{PoolOrderBook} was designed for mechanical sympathy via
pre-allocated nodes, intending to eliminate dynamic heap allocations on the critical path.
In the mixed-operation breakdown, its insert path is faster than map
(298.8~ns vs 420.4~ns), but it remains slower than array (46.1~ns) and shows a
high match latency (121.1~ns), which is over double the cost of the 
standard \texttt{MapOrderBook} (59.5~ns) in this scenario. 
However, as shown in Table~\ref{tab:direct_key_latency_ci}, the \texttt{PoolOrderBook}
achieves a lower total mean latency (\texttt{601.28~ns}) than the map (\texttt{644.26~ns}) 
in this workload. This indicates that while pooling logic introduces a specific branching penalty during matches,
its superior insertion efficiency delivers a net performance gain in mixed regimes.
The \texttt{perf} results show that the logic to manage the free-pool causes a 
7.36\% branch miss rate in the \texttt{mixed} scenario. This penalty is caused 
by the unpredictable "filling logic" in the matching cycle 
(Figure~\ref{fig:pool_match_flow} and Appendix~\ref{app:pool_branching}). 
In a mixed workload, order sizes vary significantly, which prevents the CPU 
from predicting if a match will result in a full or partial fill. As 
illustrated in Figure~\ref{fig:pool_match_flow}, when an order is fully 
consumed, the book must execute a complex path to remove the node and update 
the free-pool. If the fill is partial, this path is skipped. These outcomes in 
the \texttt{mixed} scenario occur with approximately equal frequency, leading 
to branch mispredictions that stall the CPU pipeline~\citep{patterson2013cod}.

The relative slowdown trends reinforce these architectural trade-offs. 
The \textbf{HybridOrderBook} works well in concentrated markets relative to its
deeper-book behaviour, but a tier-management penalty appears in the
\texttt{dense\_full} scenario, where hybrid reaches 2,143.29~ns versus
1,099.85~ns for map. This suggests that managing multiple 
tiers only helps when the top levels are active enough to cover the extra lookup 
costs.

Similarly, the \textbf{PoolOrderBook} shows its advantage in several dispersed
or mixed regimes. In \texttt{sparse\_extreme} and \texttt{mixed},
it outperforms map by 23.42~ns and 42.98~ns respectively. 
This performance lead is statistically supported by non-overlapping 95\% 
bootstrap confidence intervals in the mixed scenario (Pool: [590.38, 614.31]~ns 
vs.\ Map: [634.88, 654.33]~ns), and confirmed by a paired sign-flip permutation 
test ($p<0.0001$) --- formally establishing that the pooling benefit is genuine 
and not an artefact of run-to-run noise. However, 
this advantage remains marginal compared to the ArrayOrderBook 
([205.56, 215.72] ns), which supports the latter half of \ref{hyp:h2}: 
optimising allocation logic does not resolve the fundamental 
sequential bottleneck of the underlying search structure. This bottleneck 
is most pronounced in the \texttt{dense\_full} scenario, where the 
\textbf{PoolOrderBook} exhibits a higher mean latency than the standard map 
(1,224.14~ns vs 1,099.85~ns for map). This suggests that the increased 
branching penalty associated with managing the pool's free-list likely 
outweighs the locality gains in high-throughput, cache-saturated regimes.

\paragraph{Architectural Selection: No Universal Answer.}
The evidence gathered here shows that no single order book is the best choice
for every market.
Each implementation performs well under particular workload conditions and
poorly under others.
This finding has a direct practical consequence: the optimal data structure
for a given exchange is a function of that exchange's market conditions, not
an engineering absolute.
If an operator can model their expected order flow - for example through
historical data collection or Monte Carlo simulation of order arrivals, fill
rates, and cancellation frequency - they can match the implementation to
the workload.
For a venue similar to a \textbf{NASDAQ equity market}, where tick-constrained
stocks concentrate activity at a small number of near-spread price levels
\citep{harris1994minimum}, both the \texttt{ArrayOrderBook} and the
\texttt{MapOrderBook} are strong candidates.
In this dataset, array remains fastest in both \texttt{tight\_spread}
(216~ns) and \texttt{fixed\_levels} (209~ns), while map is the strongest
non-array baseline in those shallow scenarios.
In contrast, a venue resembling a \textbf{CME E-mini futures} market, where
order flow is sparse across many price levels and institutional orders arrive
with large, unpredictable sizes, would favour the \texttt{PoolOrderBook} or
\texttt{MapOrderBook} over the array, since the latter's fixed interval
requires all valid prices to be pre-allocated, making rebalancing expensive
when price ranges shift rapidly~\citep{gould2013lob}.
The \texttt{ArrayOrderBook}'s inflexibility is also a concern for venues
where participants expect continuous price freedom without tick constraints.
If a client wishes to bid or ask at any arbitrary price, the array's
pre-allocated, integer-indexed structure may impose a structural limitation
that is unacceptable in practice.
The results indicate that data structure selection is an empirical decision. 
The optimal approach is to match the implementation to the expected workload 
profile and the performance trade-offs identified in this study.

\subsection{Tail Latency Distribution and Determinism}
\label{subsec:tail_latency}
Beyond mean performance, a critical metric for evaluating a high-frequency trading 
system is the \textbf{P99 tail latency}. 
This represents the performance of the bottom 1\% of operations captured over a 
benchmark duration. 
In the zero-sum competitive environment of HFT, these worst-case outliers often 
represent the difference between a profitable execution and a severe 
financial loss. 
A single high-latency event during a period of extreme market volatility can result 
in missed liquidation opportunities or unintended exposure, potentially 
jeopardising a firm's solvency. 

The primary trend observed in Table~\ref{tab:direct_p99_latency} is that the 
\texttt{ArrayOrderBook} provides the most deterministic performance profile. 
Its P99 latency remains relatively stable across both \texttt{mixed} (246~ns) 
and \texttt{dense} (231~ns) workloads, demonstrating that increasing book 
complexity does not impact its execution path. 
This stability sets the "gold standard" for \ref{hyp:h2}, indicating that a simpler 
linear layout is inherently more predictable. 
In contrast, every other implementation sees much larger tails under pressure, 
with mixed-scenario P99 values already above 1,000~ns in map, hybrid, and pool. 

The most significant regression is found in the \texttt{VectorOrderBook}, which 
exhibits a severe tail expansion from 2,633~ns in \texttt{mixed} to 
49,854~ns in \texttt{dense\_full} (a 18.9x increase). 
As previously discussed, this behaviour is a direct consequence of the 
requirement to continuously shift elements in memory during insertion and 
frequently resize the underlying container, leading to devastating 
performance costs when the book is deep. 

The \texttt{PoolOrderBook} also exhibits a distinct tail profile. 
In the \texttt{mixed} scenario, the \texttt{PoolOrderBook} achieves a 
significantly lower tail latency than the standard \texttt{MapOrderBook} 
(1,152~ns vs 2,573~ns). This finding is particularly significant: while the 
custom memory pool reduces allocation jitter by eliminating dynamic heap 
operations, the additional logic required for free-list management 
introduces branching penalties (as seen in Section~\ref{subsec:pool_hybrid_mech}) 
that prevent it from matching the $O(1)$ stability of the array. As market 
depth increases, however, the physical benefits of contiguous memory nodes 
begin to outweigh the logical cost of branching, allowing the pool to shine 
even in its worst-case operations.

The significant reduction in P99 latency observed in the \texttt{mixed} 
scenario (1,152 ns vs 2,573 ns for Map) provides primary empirical 
verification for the claim that pooling reduces jitter. By eliminating the 
costs of the kernel's allocator during periods of high memory pressure, 
the \texttt{PoolOrderBook} compresses the tail of the distribution, 
favouring deterministic execution over raw average speed. This supports 
the first half of \ref{hyp:h2} while the P99 results in \texttt{dense\_full} support 
the latter: structural complexity remains the final barrier to nanosecond 
predictability.

Finally, the \texttt{HybridOrderBook} exhibits a performance ceiling in deep 
markets. 
While its tail is competitive in \texttt{mixed} (2,665~ns), it is beaten by the 
standard map in the \texttt{dense\_full} scenario (6,418~ns vs 3,158~ns). 
This is because in a 2,000-level book, the "hot storage" (limited to the top 20 
levels) is almost always exhausted, forcing the system to pay a dual tax: 
the overhead of checking the hot-tier followed by a full tree traversal in the 
cold-tier.

The \texttt{ArrayOrderBook} consistently delivered the highest throughput and

Its ability to maintain a tight P99 tail latency under load confirms that 
index-based, cache-friendly layouts are inherently faster and more deterministic 
than complex data structures. 
Conversely, the data supports \ref{hyp:h2} by showing that while the 
\texttt{PoolOrderBook} improved upon the \texttt{MapOrderBook} in specific 
high-density scenarios through contiguous memory, it could not match the array. 
The branching penalties required to manage the memory pool created a firm 
performance ceiling. 
Therefore, we accept both hypotheses: direct array indexing is the superior 
method for algorithmic matching, and memory pooling is an optimisation rather 
than a fundamental solution to data structure complexity. While 
\ref{hyp:h2} is accepted based on the overall performance in mixed and 
sparse scenarios, the \texttt{dense\_full} results provide an 
important counter-example where the branching overhead of pool management 
outweighs its allocation benefits.
This interpretation is consistent with the interval analysis above: key direct
comparisons are supported by separated 95\% bootstrap confidence intervals, which strengthens the
decision to accept \ref{hyp:h1} and \ref{hyp:h2} in this benchmark scope.

\section{Gateway Mode - End-to-End Decomposition}
\label{sec:gateway_decomposition}

To evaluate \ref{hyp:h3} (System-Level Masking), the benchmark transitioned from 
direct isolated execution to a fully networked Gateway mode. 
This mode simulates a realistic exchange architecture, where orders travel 
from a client over a TCP socket, are parsed from FIX messages, passed through 
a lock-free SPSC queue, and finally processed by the matching engine. 
To accurately identify bottlenecks across this pipeline, a "quad-timestamp" 
telemetry system was implemented (illustrated in Figure~\ref{fig:telemetry_pipeline}). 
Four exact timestamps are recorded for every order: the client capture before 
writing to the socket ($T_1$), the gateway capture upon 
receiving and parsing the message into a C++ struct ($T_2$), the engine capture 
immediately after dequeuing ($T_3$), and the final completion 
capture ($T_4$). 
By calculating the deltas between these timers, total end-to-end latency is 
decomposed into three strict buckets:
\begin{itemize}
    \item \textbf{Network (Net):} The time taken to traverse the TCP socket, the 
    operating system stack, and decode the FIX payload (\texttt{T2 - T1}).
    \item \textbf{Queue (Que):} The time spent waiting in the thread-handoff lock-free queue (\texttt{T3 - T2}).
    \item \textbf{Engine (Eng):} The pure algorithmic latency consumed by the order book logic (\texttt{T4 - T3}).
\end{itemize}

In contrast, the actual matching engine work takes up only 62~ns.
This indicates that almost all time spent processing an order in gateway
mode is consumed by the Linux network stack and TCP transport.
The data provides clear evidence for \ref{hyp:h3}.
In a realistic networked environment, the nanosecond-level gains achieved
by better order book data structures are substantially masked by the
millisecond-scale jitter of the transport layer.
This 'masking effect' is formally demonstrated by the shift in 
statistical significance between modes. In the mixed scenario under 
direct mode, the ArrayOrderBook and MapOrderBook maintain strictly 
non-overlapping 95\% bootstrap confidence intervals (separation of $\approx$434~ns, 
$p < 0.0001$). However, in Gateway mode, the end-to-end 95\% bootstrap 
intervals for the same pair (Array: [2.52, 3.06]~ms; Map: [2.46, 2.77]~ms) 
exhibit a significant overlap. When subjected to paired sign-flip 
permutation tests, this comparison yields a statistically insignificant result 
($p > 0.99$). This indicates that the observed difference is entirely 
indistinguishable from random noise when subjected to two-sided sign-flip 
permutations. This transition from a $p < 0.0001$ result to a 
$p > 0.05$ result proves that the measurement 'noise' introduced 
by the system stack is statistically larger than the 
order book's 'signal.'
These gateway-mode findings are consistent with exchange-scale evidence that
small latency advantages can still be economically meaningful under realistic
market conditions~\citep{aquilina2022armsrace}.
This is consistent with the broader tail-latency literature, where
end-to-end variance is often dominated by transport and queueing effects
rather than isolated compute kernels~\citep{dean2013tail}.
Unless the exchange implements kernel-bypass techniques (such as DPDK or Solarflare
OpenOnload)~\citep{redhat2012_openonload}, the choice of order book implementation remains a secondary
concern for overall system latency.
While the algorithmic choice is critical for isolating performance, the 
system-level bottlenecks define the practical experience for the client.

\section{MPSC Thread Contention} 
\label{sec:mpsc_contention}

To test the system's ability to scale to $n$ concurrent clients, we use 
Moodycamel's Multi-Producer Single-Consumer (MPSC) queue. By generating 
concurrent load through multiple producer threads, we can analyse how order book 
performance is impacted by thread hand-off over the L3 cache.
This isolates L3 cache coherence effects from network layer delays, directly 
adhering to the performance isolation mechanisms required by~\NFR{7}.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.8\textwidth]{../plots/mpsc/throughput_scaling.png}
    \caption{Throughput scaling under varying producer thread counts in MPSC 
    mode. Identifies the contention point where the single consumer matching 
    engine becomes the bottleneck.}
    \label{fig:mpsc_throughput}
\end{figure}

As shown in Figure~\ref{fig:mpsc_throughput}, the system throughput 
declines as producer count rises. Each plotted producer/book point combines $n=90$ 
samples across three scenarios (\texttt{tight\_spread}, \texttt{mixed}, 
\texttt{dense\_full}). For the \texttt{ArrayOrderBook}, mean throughput drops 
from 3,911,392~ops/s at 1 producer to 3,251,566~ops/s at 8 producers. 
While the isolated matching engine, servicing a single order in ~70 ns, has a 
theoretical capacity of ~14.3M~ops/s, the system's actual throughput is 
bottlenecked by the L3 cache coherence pressure and the Moodycamel queue 
handoff, as discussed in Section~\ref{sec:gateway_decomposition}. 
The 95\% bootstrap confidence intervals for these two endpoints 
([3,862,902, 3,962,478]~ops/s and [3,197,091, 3,305,031]~ops/s respectively) 
are strictly non-overlapping, and a paired sign-flip permutation test confirms 
the decline is statistically significant ($p < 0.0001$). This proves that each 
additional producer thread creates a systemic bottleneck rather than providing 
parallel speedup - a direct consequence of contention on the shared MPSC queue 
and the resulting L3 cache coherence pressure.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.8\textwidth]{../plots/mpsc/queue_vs_engine_scaling.png}
    \caption{Latency breakdown showing how queue wait times grow as more threads are added, while engine matching time stays the same.}
    \label{fig:mpsc_queue_latency}
\end{figure}

Figure~\ref{fig:mpsc_queue_latency} explains why this happens by breaking the 
latency down into queue time and engine time. 
The graph shows that the time an order spends waiting inside the queue is 
much higher than the time it takes the engine to actually process the order. 
For the array order book, the engine remains around 68--74~ns,
while queue wait rises from about 1,024,410~ns (1 producer) to
1,545,272~ns (8 producers). 
This shows that when there is high traffic, the system is primarily limited by the 
speed of the single matching core. 
Because the producers are sending orders faster than the single matching engine 
can process them, the orders simply back up in the queue.

This stability in the engine component serves as critical evidence 
for \ref{hyp:h3}. Even as the transport layer (the queue) degrades under contention, 
the underlying algorithmic processing time remains largely deterministic. 
For instance, the VectorOrderBook engine component exhibits a marginal 
decrease in mean latency from 6,760 ns to 5,047 ns as the configuration 
moves from one to two producers, remaining stable thereafter. 
This transition is consistent with frequency-scaling residency and 
the elimination of cold-start memory penalties. Under lighter loads 
(1 producer), the processor's power-management governor may transition 
the matching core into lower frequency states between arrivals. 
At saturation (2+ producers), the core remains at its maximum operational 
frequency with critical data structures kept 'warm' in the L1 cache, 
effectively avoiding the latency overhead associated with hardware 
wake-ups and main-memory fetches.

\section{Summary of Evaluation against Requirements}
\label{sec:nfr_summary}

Functional requirements for isolated metrics (\FR{2}) and gateway 
tracking (\FR{3}) were met using the quad-timestamp design. 
Identical test scenarios ensured fairness (\NFR{2}), while code coverage confirmed 
semantic parity (\NFR{1}). These results indicate that index-based layouts are 
faster in direct mode (\NFR{3}), while gateway mode demonstrates the 
system-level masking effect expected under~\NFR{4}. Specifically, 
despite the clear architectural separation between implementations in 
direct mode ($p<0.0001$), the introduction of Transport and Queuing 
latencies causes them to become statistically indistinguishable ($p>0.05$) 
at the end-to-end level.

\section{Threats to Validity}
\label{sec:threats_to_validity}

While the benchmarks provide strong evidence, certain limitations must be noted 
to bound the validity of these results. 

\begin{itemize}
    \item \textbf{Virtualisation Overhead.}
        The use of WSL2 introduces a virtualisation tax due to the underlying 
        Hyper-V utility VM architecture~\citep{microsoft_wsl2_arch}. 
        While this environment adds inconsistent network jitter and I/O 
        overhead, the findings remain valid because 
        the penalty is applied equally across all implementations. 

    \item \textbf{Hardware Specificity.}
        The benchmarks were conducted on a Zen 4 AMD Ryzen 7600X 
        microarchitecture. 
        The specific cache sizes and branch predictor behaviour of this processor 
        directly influence the recorded latency. 
        While the relative performance trends are likely to hold across modern 
        CPUs, the exact gaps may vary on server-grade Intel or AMD hardware. 

    \item \textbf{Synthetic Workloads.}
        The scenarios use randomised order flows that do not perfectly capture 
        the correlated bursts and strategic activity of a live market. 
        In a real-world exchange, clusters of activity at specific price levels 
        might stress the data structures in ways not captured by a uniform 
        random generator. 

    \item \textbf{Loopback Interface Limitations.}
        The Gateway mode uses the \texttt{127.0.0.1} interface, which 
        captures OS kernel overhead but lacks the physical delay of a 
        real network. In a production environment with discrete hardware, 
        latencies would likely be higher and more variable due to switch 
        buffering and NIC DMA processes.

    \item \textbf{CPU and Cache Contention.}
        In Gateway and MPSC modes, the producer (client) and consumer 
        (engine) share the same physical CPU socket. Despite thread-pinning, 
        there is inevitable contention for shared resources such as the 
        \textbf{L3 Cache} and the memory controller. This may slightly 
        inflate the recorded latencies compared to a separated production 
        environment where the engine is isolated on a dedicated server.

    \item \textbf{Statistical Sample Size.}
        We reran each configuration multiple times (typically $n=30$ in direct
        mode and $n=20$ in gateway mode). This supports bootstrap confidence intervals for
        the main hypothesis checks. The limitation here is that we did not
        statistically test every implementation/scenario combination, so small
        gaps between middle-performing books should be
        treated as indicative rather than definitive.
\end{itemize}

Despite these limitations, the evidence gathered in this chapter provides a 
consistent and clear evaluation of the trade-offs between order book 
implementations.
