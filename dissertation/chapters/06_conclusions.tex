\chapter{Conclusions}
\label{chap:conclusions}

This dissertation demonstrated a C++20 exchange infrastructure for benchmarking 
order book data structures, isolating algorithmic performance from transport 
overheads. The evidence confirms that direct indexing (ArrayOrderBook, mean 
~214 ns) provides superior, deterministic performance compared to pointer-based 
alternatives. However, in gateway operation, transport overhead consumes 
88--96\% of end-to-end latency, masking these internal gains.

\section{Synthesis of Results}
The evidence supports all three research hypotheses. \textbf{\ref{hyp:h1}} is verified by 
the Array book's lead in all direct-mode scenarios. \textbf{\ref{hyp:h2}} is supported 
with the caveat that memory pooling improves jitter (Pool: 1,152 ns P99 vs Map: 2,573 ns P99) 
but remains bound by the $O(\log N)$ tree-based search of the underlying map. 
Finally, \textbf{\ref{hyp:h3}} is confirmed by the convergence of 
throughput and the high $p$-values in gateway mode, where system noise overwhelms the 
algorithmic signal.

\section{Critical Reflection}
This project provides strong evidence within a bounded experimental
scope, but should still be interpreted with care.

First, the benchmark setup captures the full software path from gateway to
engine, and the decomposition methodology gives visibility into
network latency, queue latency, and engine latency costs. However, the loopback environment does not include physical
network effects such as switch delay and NIC-level contention, so absolute
latency should be treated as a strict theoretical lower bound; in a physical 
production environment, the masking effects identified here would likely be even 
more pronounced due to hardware-level jitter.

Second, the scenario design is robust for comparing relative behaviour across
implementations, especially for highlighting dense-book stress effects. At the
same time, synthetic workloads cannot fully reproduce strategic behaviour seen
in live markets, so claims should remain bounded to the tested scenario family.

Third, the practical implication is that exchange engineers should avoid picking
data structures from microbenchmarks alone. Direct-mode results are necessary to
understand algorithmic efficiency, but gateway-mode results are necessary for
deployment decisions where system overhead dominates.

If this project were repeated, three design choices would be adjusted earlier.
First, while this study implemented rigorous paired sign-flip permutation 
tests to identify significant performance tiers, a more exhaustive 
evaluation plan would have included these formal tests for every single 
implementation-scenario combination from the first iteration. 
Second, gateway experiments would include an additional host-level networking
baseline (for example, alternative socket or
transport configurations) to separate framework effects from OS-path effects.
Third, scenario generation would be extended with longer replay windows so that
steady-state and stress-state behaviour can be contrasted more explicitly.

\subsection{Methodological Reflection}
Reflecting on the engineering process, the decision to use a custom wait-free SPSC queue
was a successful trade-off for learning and performance. However, it introduced a sec-
ondary bottleneck during timing capture. In a future iteration, implementing a dedicated,
lock-free measurement buffer with deferred storage would minimise the "observer effect"
even further. Moreover, while the Hexagonal architecture provided excellent modular-
ity for swapping order books, the cost of virtual functions in the hot path is a known
architectural tax. For applications where isolated direct-mode engine 
efficiency is the primary constraint (such as FPGA-offload co-processors or 
hardware-in-the-loop simulators), exploring CRTP (Curiously Recurring Template 
Pattern)~\citep{coplien1995crtp} for static polymorphism could reclaim 
matching-engine latency - a meaningful gain primarily in deployments where 
transport overhead has already been eliminated. This is a key consideration for 
reaching the extreme low-latency threshold in environments where the software 
stack is the primary bottleneck.

\section{Limitations}
The scope of this research is bounded by three critical dimensions. 
Architecturally, the reliance on a virtualised WSL2 environment and a 
specific consumer-grade x86 microarchitecture means that absolute latency 
figures are environment-dependent. Methodologically, the focus on a high-impact 
subset of scenario-implementation pairs means that performance nuances in 
niche market conditions remain unexplored. Most significantly, the use of a 
loopback interface for gateway benchmarking isolates the OS kernel's 
impact but omits the physical switch-latency and NIC DMA characteristics 
inherent in discrete-hardware production environments.

\section{Professional and Ethical Issues}
HFT engineering presents a significant ethical challenge. Achieving O(1) 
matching via direct indexing provides a competitive edge that deepens the 
"digital divide" in markets, as only well-capitalised firms can implement 
bespoke hardware-aware optimisations~\citep{lewis2014flash}. 
Furthermore, the prioritisation of speed can compromise safety; stripping 
validation logic for low-latency gains increases the risk of erroneous orders 
triggering "Flash Crashes"~\citep{kirilenko2017flash}. My architecture 
enforces semantic parity as a necessary safeguard. Legally, the framework 
supports compliance with MiFID II~\citep{mifid2} by allowing architects to 
quantify the latency cost of mandatory safety features and kill-switches.

\section{Practical Recommendations}
Based on the empirical findings, the following recommendations are provided for exchange architects:

\begin{itemize}
    \item \textbf{Prioritise Determinism:} Use direct-indexed layouts in latency-critical paths, as they provide the most stable matching path performance under stress.

    \item \textbf{Targeted Optimisation:} Deploy memory pooling specifically for allocation-heavy scenarios (e.g., sparse LOBs) to reduce tail jitter. While faster than the standard map in heterogeneous workloads (Mixed), pooling logic may trail it in exceptionally deep books (Dense\_Full) due to branching penalties in the matching logic.

    \item \textbf{System-Level Visibility:} Prioritise end-to-end latency decomposition. As shown in Chapter 5, internal engine gains are often masked by transport overhead, making networking the highest-impact area for further engineering investment.
\end{itemize}

\section{Future Work}
Future research should replace the POSIX stack with Kernel Bypass 
technologies (e.g., DPDK or OpenOnload) to determine if engine-level 
efficiencies become visible when transport noise is reduced. Additionally, 
offloading price-matching logic to FPGAs could eliminate the CPU jitter 
discussed in Chapter 5, potentially reaching nanosecond-scale determinism 
for institutional-grade exchange infrastructure.

\section{Closing Statement}
The research objectives were successfully met: the framework enabled 
fair benchmarking, and the decomposition evidence exposed where 
end-to-end latency is spent. Verified through sign-flip permutation 
tests and Holm-Bonferroni corrections, these findings are 
statistically robust. Data-structure engineering remains 
essential for engine efficiency, but deployment gains require 
equal attention to transport overhead.

Overall, this dissertation demonstrates that order book performance is a 
system-level problem. In isolated mode, cache-friendly indexing gives clear 
wins, but in gateway operation, transport overhead can dominate and mask those 
gains. The framework and evidence produced here provide a practical basis for 
future research and exchange design decisions. Ultimately, as the latency 
gap between participants continues to narrow, the architectural 
optimisations explored here remain a critical frontier in the 
technological "arms race" that defines modern market efficiency 
\citep{aquilina2022armsrace}.
