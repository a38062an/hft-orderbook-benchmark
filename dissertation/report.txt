Abstract
Whilealgorithmicoptimisationisacornerstoneofhigh-frequencytrading(HFT),itsprac-
tical impact is defined by the hardware-network stack. This dissertation presents a mod-
ular C++20 exchange infrastructure for benchmarking five order book implementations:
array (direct-indexed), map (associative tree), vector (sorted), pool (slab-allocated), and
hybrid (hot-cold tiered). These are evaluated in both direct and gateway modes. Three
research hypotheses are evaluated: that cache-friendly indexing outperforms pointer-
heavy alternatives (H1), that memory pooling reduces allocation costs without eliminat-
ingstructuralpenalties(H2), andthatsystem-leveltransportoverheadmasksengine-level
gains (H3). To ensure scientific rigour, each claim is verified through Welch’s t-tests with
Bonferroni corrections and 95% confidence intervals (n = 30). In direct mode, the array
implementation achieves a mean latency of 210 ns across scenarios, decisively outper-
forming all alternatives (p < 0.0001). However, in gateway mode, a quad-timestamp
decomposition reveals that transport and queueing consume 88–96% of total latency,
causing implementations to become statistically indistinguishable at the end-to-end level
(p = 0.21). These results identify where engineering effort should transition from match-
ing engine optimisation to transport-layer efficiency. All metrics are bounded to a pinned
WSL2/AMD Ryzen environment.

Chapter 1
Introduction
1.1 Context and Motivation
In high-frequency trading (HFT), firms compete to execute orders at the lowest pos-
sible latency (Aldridge, 2013; Bilokon and Gunduz, 2023). In this environment, the
order book serves as the fundamental state-machine on the critical path. The design of
this component is a system-level problem where cache-coherence, branch-prediction, and
network-stack overhead intersect to define performance (Gould et al., 2013; Thompson
et al., 2011).
Given this dependence on low-latency software, understanding the performance impli-
cations of different order-book designs is important for both researchers and practitioners.
This dissertation evaluates and compares order-book data structures in a realistic bench-
marking environment. Detailed technical definitions are provided in Chapter 2; this
chapter focuses on the research problem, objectives, and scope of the study.
1.2 Problem Statement and Research Gap
Although there has been significant work on order-book data structures, much of industry
practice remains opaque because implementation details are commercially sensitive. Any
advantage gained from an optimised order-book design can be worth substantial financial
value, so firms rarely disclose architecture decisions in detail. This creates a role for
academic benchmarking: providing a reproducible evaluation of the architectures that
drive global markets. This study addresses a methodological gap by adopting a two-
mode philosophy. While isolated performance is an important metric, it does not capture
the "Masking Effect" of the full software path. This dissertation presents a framework
that decomposes latency across direct and gateway modes to show where algorithmic
wins yield to system-level bottlenecks.
1
1.3 Aim and Objectives
This dissertation aims to design, implement, and evaluate a C++20 exchange infrastruc-
tureforbenchmarkingorderbookdatastructuresinhigh-frequencytradingenvironments.
The specific objectives are as follows:
• Define a modular benchmarking framework with interchangeable order-book imple-
mentations under a common interface.
• Implement and validate five order-book variants (map, array, vector, pool, and
hybrid) with semantic parity across shared test scenarios.
• Evaluate performance in direct mode and gateway mode to separate algorithmic
behaviour from end-to-end system overhead.
• Decompose gateway latency into network latency, queue latency, and engine latency
to localise bottlenecks.
• Interpret observed differences using data-structure mechanics and hardware-aware
reasoning grounded in measured evidence.
To evaluate these objectives, the study uses direct mode and gateway mode, and
reports mean latency, P99 latency, throughput, and component-level decomposition of
network latency, queue latency, and engine latency against H1–H3.
1.4 Research Hypotheses
The evaluation tests three hypotheses that connect algorithmic structure, allocator strat-
egy, and systems overhead:
• H1 (Data Structure and Cache Locality): index-based order-book layouts are
expected to outperform pointer-heavy alternatives in isolated operation.
• H2 (Memory Allocation Limits): memory pooling is expected to reduce se-
lectedallocationcostsbutnotremovethefundamentalstructuralpenaltiesofslower
layouts.
• H3 (System-Level Masking): in gateway operation, network and queue over-
heads are expected to compress or hide many internal data-structure performance
differences.
For clarity, H1 is treated as supported when index-based designs consistently out-
perform pointer-heavy designs in direct mode across the core scenarios; H2 is treated
2
as supported when pooling improves selected allocation-heavy cases without eliminating
structuralpenaltiesacrossmixedworkloads; andH3istreatedassupportedwhengateway
decomposition shows network and queue costs dominating engine cost, with compressed
cross-implementation differences.
These hypotheses and acceptance criteria are defined formally in Chapter 5.
1.5 Scope and Boundaries
This dissertation focuses on software-path benchmarking within a controlled host envi-
ronment, including direct and gateway execution modes, multiple workload profiles, and
comparative analysis across five order-book implementations. The scope includes func-
tional validation, throughput and latency measurement, and component-level decompo-
sition of gateway costs. The scope does not include full physical network-path modelling
(for example switch traversal and NIC hardware contention), and absolute latency values
should therefore be interpreted as environment-bounded rather than universally transfer-
able.
1.6 Contributions Overview
Thedissertationmakesfourcontributions. First,itpresentsamodular,extensibleC++20
exchange framework with interchangeable order-book implementations and a TCP gate-
way path for end-to-end benchmarking. Second, it provides a reproducible benchmarking
suite that separates direct-mode and gateway-mode behaviour while decomposing latency
and throughput across pipeline components. Third, it reports empirical evidence on the
trade-offs between cache-friendly indexing and pointer-based alternatives, including the
bounded benefit profile of memory pooling. Fourth, it demonstrates a systems-level re-
sult: in gateway operation, network and queue overheads can dominate total latency and
mask differences that are visible in isolated mode.
1.7 Dissertation Structure
Chapter 2 reviews the technical background required for the study, including market
microstructure and hardware-aware performance considerations. Chapter 3 presents the
architectural design, requirements framework, and benchmarking philosophy. Chapter 4
details implementation choices for the exchange pipeline and order-book variants. Chap-
ter 5 reports functional validation and performance results against the research hypothe-
ses. Finally, Chapter 6 summarises findings, limitations, and future work.

Chapter 2
Background
2.1 High-Frequency Trading and the Limit Order
Book
High-Frequency Trading (HFT) is a specialized form of algorithmic trading that exe-
cutes orders at millisecond or microsecond timescales. HFT strategies are characterised
by high order-to-trade ratios, exceedingly short holding periods, and a reliance on co-
located servers running directly adjacent to exchange matching engines (Aldridge, 2013).
Leading proprietary trading firms, such as Optiver and Citadel Securities, secure direct
market data feeds to bypass the latency overhead of standard retail brokers. These firms
leverage their ultra-low latency infrastructure to execute a variety of latency-sensitive
strategies (Menkveld, 2013). This market-design and speed interaction is also synthe-
sised at the overall market-structure level by O’Hara (2015). This includes spread capture
(acting as a market maker by buying at the bid and selling at the ask), avoiding adverse
selection (cancelling stale quotes before they are traded against by faster participants),
and statistical arbitrage (exploiting fleeting price discrepancies across multiple exchange
venues). As Avellaneda and Stoikov (2008) formalise, these strategies depend directly on
how quickly the limit order book can be updated and acted upon. This is because even
small delays can erode spread capture and increase adverse-selection risk.
A core technological component of this ecosystem is the Limit Order Book (LOB),
illustrated in Figure 2.1. The LOB is a centralised electronic record of all active limit
orders - instructions to buy or sell a specific quantity of an asset at a specified price or
better (Gould et al., 2013). These orders are grouped by price levels and divided into two
sides: the bid (buyers) and the ask or offer (sellers). The highest price a buyer is willing
to pay is the best bid, while the lowest price a seller will accept is the best ask. The
numerical difference between these two values is known as the spread (visually separating
the two sides in Figure 2.1), which acts as a primary indicator of market liquidity and
volatility. The midpoint between the best bid and best ask provides a theoretical “fair
4
value”fortheassetatanygivenmicrosecond. Strategicordersubmissionandcancellation
behaviour that drives spread and depth evolution is formalised by Rosu (2009).

While the official limit order book is maintained by the exchange matching engine,
trading firms are responsible for reconstructing their own internal "shadow" books. Ex-
changes broadcast market events via raw binary protocols like ITCH or FIX, and quant
firms must rapidly parse these feeds to rebuild the current book state in their own local
memory. The performance of an HFT algorithm depends entirely on how quickly this
internal shadow book can process events. This makes the underlying data structures and
software architecture paramount.
2.2 Hardware Architecture
To achieve true ultra-low latency, trading systems must be designed to cater directly to
the physical realities of the underlying microprocessor. This approach is widely known as
Mechanical Sympathy (Fowler, 2011b). The core philosophy dictates that understanding
the hardware architecture of a machine enables software to operate in harmony with it,
unlocking maximum potential throughput. Rather than treating the CPU as an abstract
resource, HFT software treats hardware features - such as cache hierarchies and processor
pipelines - as first-class design constraints.
5
2.2.1 Cache Hierarchy and Memory Access
The cache is the primary hardware component to optimise for when striving for ultra-
low latency. It relies on the concepts of temporal and spatial locality to reduce memory
operation times. Temporal locality dictates that recently accessed memory locations
are highly likely to be accessed again in the near future. Spatial locality assumes that
memory locations physically adjacent to a recently accessed address will also be needed
soon. This behaviour closely aligns with the Pareto principle, where 80% of memory
accesses are typically concentrated within just 20% of an application’s data.
To capitalise on this, the memory subsystem is organised into a rigid hierarchical
structure, as visualised in Figure 2.2. The L1 cache is the smallest storage layer but
resides physically closest to the processor core, offering extremely fast access times of
roughly 1 - 2 nanoseconds. Subsequent caches (L2 and L3) are progressively larger but
sit further away, incurring proportionately longer fetch times. At the bottom of this
hierarchy lies the main system memory (RAM). Fetching data from main memory is
significantly slower relative to the cache, routinely taking over 100 nanoseconds and
stalling the CPU pipeline entirely (Patterson and Hennessy, 2013).
When the CPU fetches data from memory, it does not retrieve individual bytes; in-
stead, it loads contiguous blocks known as cache lines (typically 64 bytes). This mecha-
nismperfectlyleveragesspatiallocality, ensuringthatcontiguousdatastructures-likeflat
arrays - can be sequentially processed with minimal misses. Conversely, non-contiguous
allocations (such as pointer chasing in tree structures) break spatial locality, forcing the
CPU to continuously stall while waiting for slow main memory fetches.

2.2.2 Branch Prediction
Modern CPUs process computation through a specific sequence of stages known as the
instruction pipeline, conceptualised in Figure 2.3. While architectures vary, a clas-
sic pipeline consists of four fundamental stages: Fetch, Decode, Execute, and Write-
back (Patterson and Hennessy, 2013). When the CPU encounters a conditional branch
(an if-else statement) during the Fetch phase, it does not actually know which path to
take until the Execute phase resolves the condition. At this decision point, the CPU has
two choices: wait for the execution to finish, or guess the outcome. Waiting would stall
the pipeline completely, wasting upward of 15 to 20 clock cycles while the CPU sits idle
doing nothing.
To avoid this massive waste of computational time, CPUs rely on a mechanism called
a branch predictor. Instead of waiting, the branch predictor guesses which path the
program will take and immediately begins speculatively fetching and decoding those
instructions. If the guess is correct, execution continues seamlessly with zero wasted
cycles. However, if the predictor guesses incorrectly, the CPU must flush all the specu-
lative, half-finished work from the pipeline and restart fetching from the correct address.
This pipeline flush incurs a severe penalty, typically costing modern architectures like the
AMD Zen 4 between 15 and 20 cycles of completely lost time (Fog, 2025).
Modern predictors, such as the TAGE (TAgged GEometric) or perceptron-based pre-
dictors built into current AMD and Intel chips, are incredibly sophisticated and routinely
achieve over 95% accuracy (Seznec and Michaud, 2006). Mathematically, the advantage
is clear: assuming a 15-cycle misprediction penalty, a 95% accurate predictor yields an
average penalty of just 0.75 cycles per branch (0.95×0+0.05×15), vastly outperforming
a strategy of always stalling for 15 cycles.
However, despite this impressive average-case mathematical advantage, unpredictable
branches are highly detrimental to ultra-low latency systems. In HFT, average perfor-
mance is secondary to worst-case tail latency (the outermost response-time percentiles,
such as the 99.9th percentile, which dictate the absolute maximum time a system takes
to react under stress) (Dean and Barroso, 2013). If complex business logic (such as an
unpredictable conditional order fill) resides on the critical path, the inevitable 5% of
mispredictions will cause unpredictable, nanosecond-scale execution spikes. Removing
unpredictable branches entirely is therefore a primary engineering goal to ensure deter-
ministic, low-variance execution.

2.3.1 Tree-Based Implementations
Historically, many order books have been implemented using tree-based structures such
as Red-Black trees (often conceptually exposed as std::map in C++) to manage price
levels. These structures offer a theoretical O(log N) time complexity for insertions, dele-
tions, and lookups. Their primary advantage is implementation simplicity, as a map
intuitively models the domain of a limit order book by directly associating a price level
(the key) with a queue of resident orders (the value). Furthermore, trees dynamically
and gracefully handle massive, arbitrary price leaps without requiring predefined price
boundaries. However, tree nodes are invariably allocated individually on the dynamic
heap. As the order book processes thousands of intertwined updates, these nodes be-
come fragmented across disparate locations in main memory. Consequently, traversing
a tree to find a specific price level results in rapid, unpredictable pointer chasing. Each
pointer dereference frequently triggers a cache miss, forcing the CPU to repeatedly stall
while waiting for main memory fetches.
2.3.2 Flat Array Structures
To explicitly combat heap fragmentation, modern HFT architectures favour contiguous,
flat structures such as pre-allocated arrays or vectors (Bilokon and Gunduz, 2023). Al-
8
though inserting an element into a sorted array incurs an O(N) worst-case complexity
(due to shifting elements to maintain chronological order), this metric is theoretically
inferior to a tree’s O(log N). However, because arrays are strictly contiguous in memory,
traversing them allows the CPU’s hardware prefetcher to eagerly load subsequent ele-
ments into the L1 cache before they are even requested (Patterson and Hennessy, 2013).
For the relatively small, localised working sets typical of an active limit order book, the
sheer speed of contiguous localised memory scans vastly outweighs the algorithmic over-
head. In practice, an O(N) cache-friendly array scan routinely outperforms an O(log N)
tree traversal for latency-critical workloads. The severe trade-off for this speed is archi-
tectural rigidity: to achieve true O(1) direct array indexing based on price, the book must
strictly map discrete fixed-tick price intervals to precise internal array offsets, sacrificing
the dynamic flexibility of a tree.
2.3.3 Memory Pooling
The final structural consideration is how memory is allocated during live trading. Stan-
dard dynamic allocation functions like std::malloc or the new operator require the
operating system to search for free heap space and occasionally request more memory
pages from the kernel. This process requires context switching and is inherently non-
deterministic. Executing a dynamic allocation on the critical trading path is therefore
fatal to predictability because it randomly introduces severe microsecond-scale execution
spikes.
As illustrated in Figure 2.4, to bypass the operating system entirely during the trading
day, high-performance systems use Memory Pools - large contiguous blocks of memory
allocated entirely during system startup. Typically, this is achieved by instructing the OS
to persistently map a massive, pre-sized memory region (commonly via the POSIX mmap
system call) capable of securely holding millions of queued objects without exhausting
system bounds (Love, 2013). At runtime, inserting a new order simply claims a pre-
allocated object from the local pool via a fast pointer bump, guaranteeing deterministic,
O(1) nanosecond execution.
From a software engineering perspective, this profound architectural change is encap-
sulated behind standard interfaces. The memory pool acts as a drop-in replacement, ex-
posing the same high-level API methods (e.g., insert, erase) as a traditional std::map,
allowing the trading engine to remain agnostic to the underlying memory mechanics.
This, however, shifts the burden entirely onto the developer, mandating manual capacity
management and the implementation of custom logic to handle aggressive, unforeseen
out-of-memory pool exhaustions.

2.4.3 Atomic Operations and Cache Coherence
Lock-free queues remove standard mutual exclusion locks (e.g., std::mutex) to avoid
the risk of threads being put to sleep by the operating system. By eliminating context
switching, they improve determinism and reduce latency spikes during high contention.
Instead, they rely on atomic machine instructions, which allow a core to update a variable
without relinquishing the CPU.
However, writing to an atomic variable requires a core to obtain exclusive ownership
of the memory location. This triggers the hardware’s MESI protocol (Patterson and
Hennessy, 2013), which forces all other cores to invalidate their local cache copies. Com-
munication across the CPU bus to synchronise these states is orders of magnitude slower
than a local L1 cache access.
Performance is further degraded by false sharing. This occurs when unrelated vari-
ables—such as the producer’s Head index and the consumer’s Tail index—accidentally
occupy the same 64-byte hardware cache line. Because the CPU manages memory in
these 64-byte blocks, cores will constantly invalidate each other’s cache even though they
are updating different indices.
To prevent this, high-performance implementations use padding and C++20 align-
ment specifiers to ensure each index resides on its own unique cache line, as demonstrated
in the following snippet:
1 struct SPSCQueue {
2 // Force separation into different 64-byte cache lines
3 alignas(64) std::atomic<size_t> head; // Pushed by Producer
4 alignas(64) std::atomic<size_t> tail; // Advanced by Consumer
5 };
The spatial management of queue indexes is a prerequisite for scaling lock-free
performance on multi-core hardware. These techniques are grounded in foundational
non-blocking algorithm theory, notably the single-producer single-consumer and multi-
producer queues formulated by Michael and Scott (1996), which define the safety
properties required for wait-free ingestion.

2.5 Related Work
Prior literature can be grouped into market-microstructure theory, systems-design prac-
tice, and performance-engineering methodology, each with different strengths and blind
spots.
Market-microstructure sources establish why order books matter economically. Har-
ris (2002), Aldridge (2013), and Menkveld (2013) explain liquidity formation, adverse
selection, and the strategic role of fast execution. Gould et al. (2013) provides a broad
empirical and modelling survey of limit-order-book dynamics. However, these works are
not primarily designed as implementation-level benchmarking studies, so they offer lim-
ited guidance on how specific in-memory data-structure choices affect nanosecond-scale
matching-path latency. Recent exchange-scale evidence by Aquilina et al. (2022) further
shows that small latency advantages can have economically significant effects.
Systems-oriented sources describe low-latency architectural patterns but are less con-
trolled from an academic benchmarking perspective. The LMAX work (Fowler, 2011b;
Thompson et al., 2011) and more recent systematic evaluations by Bilokon and Gunduz
(2023) demonstrate the practical value of single-writer flows, cache-aware patterns, and
the "boring" (deterministic) latency profiles required for HFT. Library-level artefacts such
as moodycamel (Desrochers, 2013) show strong queue throughput under synthetic con-
ditions, but they do not by themselves establish end-to-end exchange latency behaviour
across heterogeneous order-book implementations.
Performance-methodology references provide the measurement principles needed to
connect these domains. Gregg (2013) and Dean and Barroso (2013) emphasise observer-
effect control, percentile-aware reporting, and bottleneck localisation rather than relying
on single aggregate means. This is directly relevant to HFT benchmarking, where tail be-
haviour and queueing effects can change engineering decisions even when average latency
appears similar.
The main gap is therefore not a lack of discussion on HFT speed, but a lack of
reproducible, decomposition-driven comparisons that evaluate both algorithmic and end-
to-end effects in one framework. Existing literature either focuses on financial behaviour
without implementation granularity, or on low-latency engineering patterns without con-
trolled cross-structure comparison in a unified exchange-style pipeline. This dissertation
addresses that gap by benchmarking five order-book designs under semantically equiva-
lentworkloadsinbothdirectmodeandgatewaymode, thendecomposingnetwork, queue,
and engine latency to identify where masking effects occur.

3.2 Design Goals and Constraints
The primary design goal of this benchmarking suite is to minimise measurement overhead
and avoid the ‘observer effect’ (or ‘Heisenberg effect’), where the instrumentation itself
alters the system’s performance characteristics (Gregg, 2013). Ensuring that the act of
measuring latency does not significantly inflate the results is essential for maintaining the
scientific validity of the data.
Additionally, the project must navigate the technical constraints of the Linux/WSL2

environment. While this virtualised setup introduces additional jitter and overhead com-
pared to bare-metal hardware, these factors are explicitly documented and accounted
for throughout the evaluation. By acknowledging these boundaries, we ensure that our
conclusions are grounded in the specific realities of the test environment.
3.3 High-Level Architecture
Theexchangeisdesignedasamodularsystemusingthe‘PortsandAdapters’(Hexagonal)
architecture (Velepucha and Flores, 2023; Cockburn, 2005). This approach isolates the
core trading logic from external infrastructure, ensuring that the system’s performance
characteristics remain deterministic and decoupled from transport or logging overheads.
3.3.1 Architectural Pattern: Ports and Adapters
In this design, the core Matching Engine (src/core) functions as the central "hexagon"
containing the domain logic. It remains unaware of whether it is communicating with
a real network client or a simulated benchmark feed. Communication with the engine
happens through Ports - abstract interfaces that define what the engine requires from
the outside world.
The most critical port in this system is the IOrderBook interface. By defining a stable
API for order insertion, cancellation, and matching, the engine can interact with any data
structure implementation (the Adapters) without modification. This encapsulation is the
foundation of our scientific fairness (NFR-2); it allows us to swap an ArrayOrderBook
for a MapOrderBook while keeping the surrounding benchmark harness and engine logic
identical.

3.4 The IOrderBook Interface and Polymorphism
The core of our modular design is the IOrderBook interface, implemented as a C++
abstract base class (src/core/i_order_book.hpp). This interface defines the contract

(cancelOrder), and the matching logic (match).
Using a polymorphic interface ensures that each implementation is treated as a ‘black
box’ by the benchmark harness. This is critical for scientific fairness (NFR-1); it prevents
the benchmarking logic from using implementation-specific shortcuts and ensures that
every data structure is subjected to the exact same call overhead (i.e. the nanosecond-

book’s implementation ensures that our results only reflect the efficiency of the data
structures being tested.
3.5 Benchmarking Design: The Two-Mode Philoso-
phy
To validate our hypotheses scientifically, we adopt a dual-mode benchmarking philosophy
that separates algorithmic efficiency from overhead. This design is necessary so that our
measurements capture both the theoretical potential of our data structures and their
relative value in a production-like environment.
23
The Direct Mode serves as a controlled micro-benchmark. It executes the matching
engine logic synchronously within the benchmark thread, bypassing all networking and
asynchronous layers. This isolation is essential for uncovering the ‘pure’ performance
characteristics of the underlying order book, allowing us to see latency differences that
are usually hidden by the surrounding system.
The Gateway Mode provides realism by including the full TCP/IP stack (via loop-
back), FIX protocol parsing, and the asynchronous hand-off between network and engine
threads. Measuring the system in this mode is critical for exploring the Masking Effect
proposed in Hypothesis H3. As we will evaluate in Chapter 5, the constant-time over-
head of these networking layers obscures the relative performance advantages of faster
data structures. By designing our benchmark suite with both modes, we can differentiate
between a structure’s peak efficiency and its practical impact on end-to-end latency.
3.6 Concurrency and Threading Strategy
To maintain low latency, the system must separate high-latency I/O operations - such as
network packet polling, TCP/IP stack traversal, and FIX message parsing - from the core
matching engine. This multi-threaded architecture prevents slower network components
from blocking the execution of trades, which is a standard pattern for high-performance
sequence processing in industrial matching engines (Thompson et al., 2011).
At the core of the concurrency strategy is the Producer-Consumer hand-off. The
Gateway thread (Producer) handles the I/O operations described above and pushes
parsed orders into a lock-free queue, while the Matching Engine thread (Consumer) pulls
and executes them. For the standard gateway mode, we employ a Single-Producer
Single-Consumer(SPSC)queue, whichprovideswait-freepropertiesfortheengine. In
scenarios with multiple concurrent clients, the system can swap this for a Multi-Producer
Multi-Consumer (MPMC) adapter from the moodycamel library.
To ensure deterministic performance, we use CPU Pinning (Thread Affinity). By
pinning the engine thread to a specific physical core, we prevent the operating system
from migrating the process across different CPUs. This elimination of context-switching
jitter and cache-migration overhead is vital for mitigating the Observer Effect. It ensures
that our measurements represent ’bare-metal’ performance as closely as possible within
a Linux environment.

restrictions may limit the flexibility and accessibility of the exchange compared to stan-

Torigorouslyevaluatewhetherthemeasuredlatencydifferencesaretheresultofstruc-
tural design choices or merely random measurement noise, we perform formal pairwise
comparisons using Welch’s t-test (Welch, 1947). This analysis tests the null hypothesis
(H0): the assumption that there is no real performance difference between two implemen-
tations, and any observed gap is due to chance. To reject this hypothesis, we calculate
a t-statistic, which interprets the "signal" (the difference in mean latency) relative to
the "noise" (the variance or jitter in the data). A larger t-statistic indicates that the
performance gap is many times larger than the underlying randomness of the system.
Welch’s variant of the t-test is specifically selected because it does not assume equal
variances between the compared groups. In low-latency software, different algorithms
(e.g., array indexing vs. pointer-chasing in a pool) often exhibit different levels of "tail
jitter," making this robust approach necessary for scientific accuracy (Welch, 1947).
Furthermore, because we are making four separate comparisons (comparing the Array
implementation against the Map, Pool, Vector, and Hybrid books), we must account for
the "multiple comparisons problem." If we use a standard 5% significance level (α= 0.05)
for each test, the cumulative probability of finding a "false positive" by pure luck increases
acrossthefourtests. Tomaintainastrictoverallconfidencelevel, weapplytheBonferroni
correction (Dunn, 1961). This involves dividing our target significance (α= 0.05) by the
number of comparisons (m = 4), resulting in a more conservative adjusted threshold of
αadj = 0.0125.
Across all primary scenarios, the results provide overwhelming evidence against the
null hypothesis. In the mixed regime, comparing the ArrayOrderBook to the MapOrder-
Book yields a t-statistic of t=-75.09 (df = 43.87, p<0.0001). This value indicates that
the "signal" of the array implementation’s superiority is 75 times stronger than the mea-
surement "noise." A similarly extreme separation is observed in the dense_full scenario
against the map alternative (t=-119.14, df = 30.55, p<0.0001). Even in the highest-
performing non-array tier (tight_spread), the array implementation maintains a clear
statistical lead (t=-44.92, p < 0.0001). This formal analysis confirms that the memory-

5.4.1 Architectural Mechanisms: Pool and Hybrid
Two novel implementations, the HybridOrderBook and the PoolOrderBook, were
engineered in an attempt to mitigate the architectural limitations of the standard data
structures. Rather than relying on a single collection type, these order books employ
more complex internal logic to optimise for specific hardware characteristics.
The HybridOrderBook was designed as an improvement over the standard map
by employing a tiered architecture: a highly optimised "hot storage" and a slower "cold
storage." It aims to achieve the unbounded flexibility of a map - removing the strict price-
range limitations of the array - while mimicking array-like performance for the most ac-
tively traded prices. The hot storage is implemented as a capacity-bounded std::vector
capped at the top 20 price levels. Because the vector never exceeds 20 elements, memory-
shiftingduringinsertionsispracticallyinstantaneousandfitsentirelywithintheL1cache.
The data supports this design trade-off. As seen in Table 5.6, the Hybrid book is still
slower than the ArrayOrderBook in this mixed operation breakdown (for example,
71.3 ns vs 31.7 ns on lookup), and it is also slower than MapOrderBook in all four
mixed operations shown in the same table. In other words, the expected hot-storage ad-
vantage is not observed as a net win over MapOrderBook in this dataset. A key reason
is that in shallow books, MapOrderBook traversal depth is already low. Consequently,
the overhead of hybrid’s internal routing (checking tiers and handling boundary logic)
46
outweighs the memory benefit of the 20-level hot tier. This also helps explain the higher
run-to-run spread observed for hybrid in mixed and high_cancellation: cancellation-
heavyflowsandheterogeneousorderplacementincreasetier-crossingfrequency, sobranch
outcomes in the routing path become less predictable on average. However, its perfor-
mance is sensitive to market depth. In shallow scenarios, HybridOrderBook remains
in the same non-array performance band as MapOrderBookand PoolOrderBook(for
example, tight_spread: 678 vs 639 vs 596 ns; fixed_levels: 689 vs 664 vs 609 ns), but
in the dense_full scenario its mean latency degrades to 2,143.29 ns. As market depth
expands, a majority of orders are pushed out of the 20-level vector and into the slower
cold std::map, subjecting them to tree-traversal overhead.
In contrast, the PoolOrderBook was designed for mechanical sympathy via pre-
allocated nodes, intending to eliminate dynamic heap allocations on the critical path. In
the mixed-operation breakdown, its insert path is faster than map (298.8 ns vs 420.4 ns),
but it remains slower than array (46.1 ns) and shows a high match latency (121.1 ns).
The perf results show that the logic to manage the free-pool causes a 7.36% branch miss
rate in the mixed scenario. This penalty is caused by the unpredictable "filling logic" in
the matching cycle (Figure 5.7 and Appendix F). In a mixed workload, order sizes vary
significantly, which prevents the CPU from predicting if a match will result in a full or
partial fill. As illustrated in Figure 5.7, when an order is fully consumed, the book must
execute a complex path to remove the node and update the free-pool. If the fill is partial,
this path is skipped. These outcomes in the mixed scenario occur with approximately
equal frequency, leading to branch mispredictions that stall the CPU pipeline (Fog, 2025).

5.4.2 Tail Latency Distribution and Determinism
Beyond mean performance, a critical metric for evaluating a high-frequency trading sys-
tem is the P99 tail latency. This represents the performance of the bottom 1% of op-
erations captured over a benchmark duration. In the zero-sum competitive environment
of HFT, these worst-case outliers often represent the difference between a profitable exe-
cution and a severe financial loss. A single high-latency event during a period of extreme
49
market volatility can result in missed liquidation opportunities or unintended exposure,
potentially jeopardizing a firm’s solvency.
The primary trend observed in Table 5.7 is that the ArrayOrderBook provides the
most deterministic performance profile. Its P99 latency remains relatively stable across
both mixed (246 ns) and dense (231 ns) workloads, demonstrating that increasing book
complexity does not impact its execution path. This stability sets the "gold standard" for
H2, indicating that a simpler linear layout is inherently more predictable. In contrast,
every other implementation sees much larger tails under pressure, with mixed-scenario
P99 values already above 1,000 ns in map, hybrid, and pool.
The most significant regression is found in the VectorOrderBook, which exhibits a se-
vere tail expansion from 2,633 ns in mixed to 49,854 ns in dense_full (a 18.9x increase).
As previously discussed, this behaviour is a direct consequence of the requirement to con-
tinuously shift elements in memory during insertion and frequently resize the underlying
container, leading to devastating performance costs when the book is deep.
The PoolOrderBook also exhibits a distinct tail profile. In the mixed scenario, it per-
forms significantly worse than the array (1,152 ns vs 246 ns) due to the high frequency
of branch mispredictions documented in Section 5.4.1. This data provides a direct vali-
dation of H2: although memory pooling reduces allocation jitter, it cannot overcome the
fundamental performance bottlenecks (in this case, branching stalls) of a more logically
complex data structure. However, in the dense_full case, the pool’s P99 improves rel-
ative to map (3,096 ns vs 3,158 ns for map). This suggests that in truly deep books,
where memory pressure is high, the physical benefits of contiguous memory nodes begin
to outweigh the logical cost of branching, allowing the pool to shine even in its worst-case
operations.
Finally, the HybridOrderBook exhibits a performance ceiling in deep markets. While
its tail is competitive in mixed (2,665 ns), it is beaten by the standard map in the
dense_full scenario (6,418 ns vs 3,158 ns). This is because in a 2,000-level book, the
"hot storage" (limited to the top 20 levels) is almost always exhausted, forcing the system
to pay a dual tax: the overhead of checking the hot-tier followed by a full tree traversal
in the cold-tier.

delays, directly adhering to the performance isolation mechanisms required by NFR-7.

5.7 Summary of Evaluation against Requirements
Functional requirements for isolated metrics (FR-2) and gateway tracking (FR-3) were
met using the triple-timer design. Identical test scenarios ensured fairness (NFR-2), while
code coverage confirmed semantic parity (NFR-1). These results indicate that index-
based layouts are faster in direct mode (NFR-3), while gateway mode demonstrates the
system-level masking effect expected under NFR-4.
5.8 Threats to Validity
While the benchmarks provide strong evidence, certain limitations must be noted to
bound the validity of these results.
• Virtualisation Overhead. The use of WSL2 introduces a virtualisation tax due
to the underlying Hyper-V utility VM architecture (Microsoft, 2024). While this
environment adds inconsistent network jitter and I/O overhead, the findings remain
valid because the penalty is applied equally across all implementations.
• Hardware Specificity. The benchmarks were conducted on a Zen 4 AMD Ryzen
7600X microarchitecture. The specific cache sizes and branch predictor behaviour
57
of this processor directly influence the recorded latency. While the relative perfor-
mance trends are likely to hold across modern CPUs, the exact gaps may vary on
server-grade Intel or AMD hardware.
• Synthetic Workloads. The scenarios use randomized order flows that do not
perfectly capture the correlated bursts and strategic activity of a live market. In
a real-world exchange, clusters of activity at specific price levels might stress the
data structures in ways not captured by a uniform random generator.
• Loopback Interface Limitations. The Gateway mode uses the 127.0.0.1 in-
terface, which captures OS kernel overhead but lacks the physical delay of a real
network. Inaproductionenvironmentwithdiscretehardware, latencieswouldlikely
be higher and more variable due to switch buffering and NIC DMA processes.
• CPU and Cache Contention. In Gateway and MPSC modes, the producer
(client)andconsumer(engine)sharethesamephysicalCPUsocket. Despitethread-
pinning, there is inevitable contention for shared resources such as the L3 Cache
and the memory controller. This may slightly inflate the recorded latencies com-
pared to a separated production environment where the engine is isolated on a
dedicated server.
• Statistical Sample Size. We reran each configuration multiple times (typically
n = 30 in direct mode and n = 20 in gateway mode). This supports confidence
intervals for the main hypothesis checks. The limitation here is that we did not
statisticallytesteveryimplementation/scenariocombination, sosmallgapsbetween
middle-performing books should be treated as indicative rather than definitive.
Despite these limitations, the evidence gathered in this chapter provides a consistent
and clear evaluation of the trade-offs between order book implementations.

Conclusions
6.1 Summary of Contributions
This dissertation demonstrated the design and implementation of a C++20 exchange
infrastructure for benchmarking order book data structures. It explored the trade-offs
to consider when designing order books as well as the implications of real-world system
overheads. The key contributions of this work are as follows:
• A modular C++20 exchange framework with interchangeable order book imple-
mentations and a TCP gateway for end-to-end benchmarking.
• A benchmarking suite that decomposes latency and throughput across direct and
gateway modes, isolating data structure performance from system overheads.
• Empirical evidence on the performance trade-offs of direct indexing versus pointer-
based order book strategies, and the limited impact of memory pooling on overall
latency.
• A clear systems-level finding that in end-to-end gateway operation, network and
transport overheads dominate latency, masking internal data structure differences
that are visible in isolated direct mode.
These contributions advance our understanding of how order book design choices in-
teract with system-level factors to influence performance in high frequency trading envi-
ronments. The modular framework and comprehensive benchmarks provide a foundation
for future research and practical exchange design decisions.
6.2 Answers to Research Hypotheses
In terms of hypotheses H1–H3, the evidence supports the following conclusions. These
conclusions are based on the repeated-run results in Chapter 5, with confidence intervals
59
and gateway latency breakdown where relevant.
H1: Direct indexing outperforms pointer-heavy alternatives in isolated mode.
Status: Supported.
Direct-mode benchmarks show that the direct-indexed array outperforms alternative
structures across almost all scenarios, with mean latency around 210 ns. Results also
indicate that non-array structures remain clearly slower than array even in shallow sce-
narios, reinforcing H1. This is reflected consistently in the direct-mode results (Table 5.3
and Table 5.6).
H2: Memory pooling reduces allocation jitter but does not remove structural
complexity costs. Status: Supported with Caveats.
Results indicate that memory pooling improves performance in selected scenarios,
particularly sparse_extreme and mixed. At the same time, the pool logic introduces
its own structural cost in mixed workloads, where branch misprediction and match-path
complexity increase latency, and in dense_full pool still trails map. So pooling is clearly
useful, but it is an optimisation, not a full solution to data-structure complexity. This
trade-off is discussed directly in Section 5.4.1.
H3: In end-to-end gateway operation, system overhead masks internal data-
structure gains. Status: Supported. In gateway mode, network latency and queue
latency dominate total latency, while engine latency remains very small in comparison.
For example, for the Array book the network component is 2,014,045 ns out of 2,100,661
ns total, showing that most end-to-end cost is outside the matching algorithm itself. This
transition from a p < 0.0001 result to a p > 0.05 result proves that the measurement

This explains why differences visible in direct mode are often compressed in gateway
mode, while still allowing scenario-specific exceptions (notably hybrid in dense_full)
where queue/network accumulation diverges. This pattern is captured in Table 5.8.
6.3 Critical Reflection
This project provides strong evidence within a bounded experimental scope, but should
still be interpreted with care.
First, the benchmark setup captures the full software path from gateway to engine,
and the decomposition methodology gives visibility into network latency, queue latency,
and engine latency costs. However, the loopback environment does not include physical
network effects such as switch delay and NIC-level contention, so absolute latency should
be treated as a strict theoretical lower bound; in a physical production environment, the
60
masking effects identified here would likely be even more pronounced due to hardware-
level jitter.
Second, the scenario design is robust for comparing relative behaviour across im-
plementations, especially for highlighting dense-book stress effects. At the same time,
synthetic workloads cannot fully reproduce strategic behaviour seen in live markets, so
claims should remain bounded to the tested scenario family.
Third, the practical implication is that exchange engineers should avoid picking data
structures from microbenchmarks alone. Direct-mode results are necessary to understand
algorithmic efficiency, but gateway-mode results are necessary for deployment decisions
where system overhead dominates.
If this project were repeated, three design choices would be adjusted earlier. First,
while this study implemented rigorous pairwise Welch’s t-tests to identify significant per-
formance tiers, a more exhaustive evaluation plan would have included these formal tests
for every single implementation-scenario combination from the first iteration. This would
integrate statistical rigour even more deeply into the data-gathering phase. Second, gate-
way experiments would include an additional host-level networking baseline (for example,
alternativesocketortransportconfigurations)toseparateframeworkeffectsfromOS-path
effects. Third, scenario generation would be extended with longer replay windows so that
steady-state and stress-state behaviour can be contrasted more explicitly.
6.3.1 Methodological Reflection
Reflecting on the engineering process, the decision to use a custom wait-free SPSC queue
was a successful trade-off for learning and performance. However, it introduced a sec-
ondarybottleneckduringtimingcapture. Inafutureiteration, implementingadedicated,
lock-free measurement buffer with deferred storage would minimise the "observer effect"
even further. Moreover, while the Hexagonal architecture provided excellent modular-
ity for swapping order books, the cost of virtual functions in the hot path is a known
architectural tax. Exploring CRTP (Curiously Recurring Template Pattern) for static
polymorphism could potentially reclaim several nanoseconds of matching-engine latency.
This is a key consideration for reaching the extreme low-latency threshold.
6.4 Limitations
• Platform dependency: results were measured on a pinned WSL2/Zen4 host, so
absolute numbers may change on other operating-system and CPU stacks.
• Workload realism: scenarios are representative and diverse, but still synthetic, and
therefore cannot capture every market microstructure effect.
61
• Statistical scope: this dissertation uses confidence intervals and pairwise Welch’s
t-tests for the main hypothesis checks, but does not exhaustively test every
implementation-scenario combination.
• Loopback boundary: gateway mode exercises software networking cost but excludes
physical network path variability.
6.5 Professional, Ethical, Social, and Legal Issues
Developing sub-microsecond matching engines is as much an ethical challenge as a tech-
nical one. As this dissertation demonstrates, achieving O(1) matching via the Array-
OrderBook provides a massive competitive edge. This "race to zero" creates a signifi-
cant "digital divide" in the markets; only large institutional players with the capital to
implement hardware-aware optimisations and co-located infrastructure can realistically
compete. Smaller firms and retail investors, who cannot afford isolated CPU cores and
bespoke memory management, are left at a structural disadvantage (Lewis, 2014).
There is also a clear conflict between raw speed and market stability. Stripping
away branching logic and validation checks to reduce latency, as I explored in Chapter 4,
introduces systemic risk. Algorithms that prioritise throughput over safety can propagate
erroneous orders at nanosecond scales, potentially triggering "Flash Crashes" that human
traders cannot stop in time (Kirilenko et al., 2017). To counter this, the architecture
built here enforces semantic parity and boundary checks—not just as good engineering,
but as a necessary ethical safeguard.
Legally, modern exchange infra must sit within the MiFID II regulatory framework
(European Union, 2014). This requires "kill-switches" and robust testing prior to deploy-
ment. My benchmarking suite acts as a key compliance tool here; it allows engineers to
measure the exact latency "tax" of these mandatory safety features, ensuring the engine
remains stable and compliant without flying blind into production.
6.6 Practical Recommendations
Basedontheempiricalfindings, thefollowingrecommendationsareprovidedforexchange
architects:
• Prioritise Determinism: Use direct-indexed layouts in latency-critical paths, as
they provide the most stable Matching-path performance under stress.
• TargetedOptimisation: Deploymemorypoolingspecificallyforallocation-heavy
scenarios (e.g., Sparse LOBs), as the structural overhead of pooling logic may de-
grade performance in denser scenarios.
62
• System-Level Visibility: Prioritise end-to-end latency decomposition. As shown
in Chapter 5, internal engine gains are often masked by transport overhead, making
networking the highest-impact area for further engineering investment.
6.7 Future Work
First, the ’masking effect’ identified in gateway-mode benchmarks suggests that the next
logical step is to move beyond the POSIX stack. Future research should integrate Kernel
Bypass technologies, such as DPDK (Data Plane Development Kit) or user-space trans-
port like OpenOnload. By eliminating the OS kernel from the hot path, researchers can
evaluate whether the engine-level efficiencies identified in direct mode become visible at
the end-to-end level.
Second, the stability of the ArrayOrderBook suggests it is a primary candidate for
Hardware Acceleration. Implementing the price-map logic on an FPGA (Field Pro-
grammable Gate Array) would remove the CPU jitter and frequency-scaling penalties
discussed in Chapter 5. A hybrid hardware-software architecture could offload the Price-
TimeprioritymatchingtotheFPGAwhilemaintainingtheFIXprotocollogicinC++20,
potentially reaching nanosecond-scale determinism across all market scenarios.
6.8 Closing Statement
Taken together, the objectives set in Chapter 1 were met: the framework enabled fair
cross-implementation benchmarking, direct mode and gateway mode exposed different
bottlenecks, and the decomposition evidence showed where end-to-end latency is actu-
ally spent. To ensure these findings were mathematically sound, every quantitative claim
was verified through a formal statistical framework incorporating Welch’s t-tests and
Bonferroni corrections. This rigour demonstrates that the observed performance trends
are scientifically robust and not attributable to measurement noise. The practical im-
plication is that data-structure engineering remains essential for engine efficiency, but
deployment-level latency improvements require equal attention to transport and queue-
ing overhead.
Overall, this dissertation demonstrates that order book performance is a system-level
problem. In isolated mode, cache-friendly indexing gives clear wins, but in gateway
operation, transport overhead can dominate and mask those gains. The framework and
evidence produced here provide a practical basis for future research and exchange design
decisions.
